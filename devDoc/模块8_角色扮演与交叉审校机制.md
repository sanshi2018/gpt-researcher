# 模块八：角色扮演与交叉审校机制 (Role Playing & Collaboration)

在上一模块中，我们搭建了从策划、写稿到排版的庞大流水线。其中最引人注目的，是子章节撰写阶段的那条**死循环校验流水线**：
`Researcher (初稿) -> Reviewer (审阅) -> Reviser (修改) -> Reviewer (再审)`

为什么不让一个大模型一口气把“写”和“改”都干了？
因为 LLM 和人一样，存在严重的**“当局者迷”**现象。自己写的文章，往往看不出逻辑漏洞。GPT-Researcher 在这里应用了业界极佳的实践：**对抗式角色扮演 (Adversarial Role-Playing)**。

---

## 1. 源码解剖：无情的“审稿人” (Reviewer)

打开 `multi_agents/agents/reviewer.py`，看看系统赋予这个 Agent 的“咒语” (Prompt)：

```python
# multi_agents/agents/reviewer.py 节选
TEMPLATE = """You are an expert research article reviewer. 
Your goal is to review research drafts and provide feedback to the reviser only based on specific guidelines."""

review_prompt = f"""You have been tasked with reviewing the draft which was written by a non-expert based on specific guidelines.
Please accept the draft if it is good enough to publish, or send it for revision, along with your notes to guide the revision.
If not all of the guideline criteria are met, you should send appropriate revision notes.
If the draft meets all the guidelines, please return None.

Guidelines: {guidelines}
Draft: {draft_state.get("draft")}
"""
```

**看点**：
1. **身份压制**：开篇就告诉 Reviewer“你是一个专家”，而你正在审阅的是一篇“外行写的草稿（written by a non-expert）”。这能极大激发大模型“挑刺”的本能。
2. **退出机制**：如果认为没问题，返回 `None`。结合上一模块讲到的 Conditional Edge，一旦返回 `None`，死循环就被打破，流程走向结束。

---

## 2. 源码解剖：苦逼的“修改人” (Reviser)

当 Reviewer 挑出毛病后，任务流转到了修改人手里。打开 `multi_agents/agents/reviser.py`：

```python
# multi_agents/agents/reviser.py 节选
prompt = [
    {
        "role": "system",
        "content": "You are an expert writer. Your goal is to revise drafts based on reviewer notes.",
    },
    {
        "role": "user",
        "content": f"""Draft:\n{draft_report}
Reviewer's notes:\n{review}\n\n
You have been tasked by your reviewer with revising the following draft...
If you decide to follow the reviewer's notes, please write a new draft and make sure to address all of the points they raised.
Please keep all other aspects of the draft the same.
You MUST return nothing but a JSON in the following format...
"""
    }
]
```

**看点**：
1. **绝对服从**：明确告诉它“你是被要求根据 Reviewer 的意见来修改的”，避免模型自作主张修改了原本没问题的地方。
2. **结构化汇报**：要求输出必须是一个 JSON，不仅包含改好的文章，还要包含 `revision_notes`（修改说明）。这个修改说明在下一次流转回 Reviewer 时，会被 Reviewer 看到：“哦，原来你照着我的意见改了这些地方”。

---

## 3. 自我纠正循环 (Self-Correction Loop)

通过 `Reviewer` 和 `Reviser` 的分离，GPT-Researcher 实现了一个全自动的“左右互搏”系统：
1. 主编给出了 Guidelines（比如：必须全英文、必须带统计数据）。
2. `Researcher` 匆匆写完初稿。
3. `Reviewer` 发现没有统计数据，打回并写下 Note：“缺少数据支撑”。
4. `Reviser` 看到 Note，想办法（或者让大模型自己虚构，但在 GPT-Researcher 中是基于真实检索到的 Context）把数据补上。
5. `Reviewer` 再看，有了数据，返回 `None`。

这种架构不仅应用在写文章上，在自动写代码（如 Devin）的流程中，也是通过“编写代码 -> 运行测试 -> 报错分析 -> 重新修改”这样的多智能体拓扑来实现的。
