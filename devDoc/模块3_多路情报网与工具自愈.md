# 模块三：多路情报网与工具自愈 (Web Scraping & Tool Routing)

在上一模块中，我们的“主编”通过 LLM 拆解出了具体的搜索关键词（工单）。但要真正写出一篇言之有物的报告，必须派“情报员”去互联网上捞取海量的事实数据。

这就是 GPT-Researcher 架构中极为核心的两个底层组件：**检索器 (Retrievers)** 与 **爬虫 (Scrapers)**。

---

## 1. 架构设计：两段式情报收集

GPT-Researcher 对外部信息的获取采用了经典的“两段式”架构：

1. **广撒网的检索 (Retrieve)**：利用各大搜索引擎 API（Tavily, Google, DuckDuckGo）快速搜寻跟 Query 相关的网页链接与简要摘要（Snippet）。
2. **深挖洞的爬取 (Scrape)**：拿着搜索到的链接，逐个进入网页抓取正文全文。

这种设计的精妙之处在于**高度解耦**。无论底层是用 Google 还是 Bing 搜索，最后都会转化为统一的结构（`href` 和 `body`），再交给下游的爬虫去处理。

---

## 2. 源码解剖：动态路由与异常兜底

### 检索器 (Retrievers)：多路情报网
打开 `gpt_researcher/retrievers/__init__.py`，你会看到琳琅满目的搜索源：
*   **通用引擎**：Google, Bing, DuckDuckGo, Searx
*   **专业引擎**：Arxiv (论文), PubMed (医学), SemanticScholar
*   **AI 专属 API**：Tavily (专为 LLM 设计的搜索引擎), Exa, Firecrawl

**多路召回逻辑**：在 `actions/agent_creator.py` 或者 `config` 中，你可以配置同时使用多个检索器。这意味着一次查询，可以同时去 Google 和医学库里捞数据，做到“兼听则明”。

### 爬虫 (Scrapers)：动态工具路由
找到网页链接后，如何抓取内容？打开 `gpt_researcher/scraper/scraper.py`，看这个核心方法 `get_scraper`：

```python
# gpt_researcher/scraper/scraper.py 节选
def get_scraper(self, link):
    SCRAPER_CLASSES = {
        "pdf": PyMuPDFScraper,
        "arxiv": ArxivScraper,
        "bs": BeautifulSoupScraper,
        "browser": BrowserScraper,
        "tavily_extract": TavilyExtract,
        ...
    }

    if link.endswith(".pdf"):
        scraper_key = "pdf"
    elif "arxiv.org" in link:
        scraper_key = "arxiv"
    else:
        scraper_key = self.scraper # 用户配置的默认爬虫

    return SCRAPER_CLASSES.get(scraper_key)
```
**看点 (Tool Routing)**：这就像是一个智能的路由器。如果碰到以 `.pdf` 结尾的链接，它会自动切换到 `PyMuPDF` 解析器；如果发现是论文网站 `arxiv.org`，自动换成专门处理论文排版的爬虫；普通的网页就用默认的 `BeautifulSoup` 或 `Playwright` 浏览器无头模式。

### 异常自愈与兜底 (Self-Correction)
在爬虫遍历网页时，经常会遇到 404、反爬虫拦截或者干脆抓出来全是乱码。源码中的防御机制非常值得学习：
```python
# scraper.py 中的错误处理
try:
    content, image_urls, title = await scraper.scrape_async()
    if len(content) < 100:
        self.logger.warning(f"Content too short or empty for {link}")
        return {"url": link, "raw_content": None, ...} # 剔除无效内容
except Exception as e:
    self.logger.error(f"Error processing {link}: {str(e)}")
    return {"url": link, "raw_content": None, "image_urls": [], "title": ""}
```
即便某个网页挂了，它只会抛出警告，并返回 `None` 占位，**绝对不会导致整个 Agent 崩溃**。下游在合并数据时，会自动过滤掉这些 `None`。

## 3. 实操体验：编写独立的爬虫工具链 (MRE)

让我们抛开 Agent，单独调用底层的 Scraper 模块，体验它是如何智能识别 URL 并并发爬取内容的。

### MRE 代码：智能爬虫分发器
在根目录下新建文件 `mre_module3.py`：

```python
import asyncio
from gpt_researcher.scraper.scraper import Scraper
from gpt_researcher.utils.workers import WorkerPool

async def main():
    # 模拟三个不同类型的链接：一个是普通网页，一个是 Arxiv 论文页面，一个是 PDF
    urls = [
        "https://arxiv.org/abs/1706.03762",  # Transformer 经典论文
        "https://www.w3.org/WAI/ER/tests/xhtml/testfiles/resources/pdf/dummy.pdf", # 虚拟 PDF 测试文件
        "https://www.openai.com"             # 普通网站
    ]
    
    print(f"🚀 准备爬取以下链接，系统会自动分配对应的爬虫工具：")
    for u in urls:
        print(f" - {u}")
        
    print("\n⏳ 正在并发爬取，请稍候...")
    
    # 1. 实例化 WorkerPool 控制并发
    worker_pool = WorkerPool(max_workers=5)
    
    # 2. 实例化全局 Scraper
    # 我们指定默认的基础爬虫为 bs (BeautifulSoup)
    master_scraper = Scraper(
        urls=urls,
        user_agent="Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36",
        scraper="bs",
        worker_pool=worker_pool
    )
    
    # 3. 运行爬虫
    results = await master_scraper.run()
    
    print("\n✅ 爬取完成！结果摘要：")
    for res in results:
        title = res.get("title", "Unknown Title")
        content = res.get("raw_content", "")
        url = res.get("url", "")
        # 如果爬取失败，内容可能为空
        content_len = len(content) if content else 0
        
        print("-" * 50)
        print(f"🔗 URL: {url}")
        print(f"📌 标题: {title}")
        print(f"📏 抓取字数: {content_len} 字符")
        if content_len > 0:
            print(f"📄 节选: {content[:100]}...")

if __name__ == "__main__":
    asyncio.run(main())
```

### 运行结果预期
运行这个脚本（`python mre_module3.py`），你会看到系统自动发起了异步并发请求。
对于 Arxiv，它启动了专用的 arXiv API 去抓取摘要和作者；对于 PDF，它默默调用了 `PyMuPDF` 提取文本；对于 OpenAI 官网，它使用了普通的 `BeautifulSoup` 去除了 HTML 标签，留下了纯净的正文内容。

## 下一步 (Next Steps)

现在，我们通过爬虫从四面八方抓来了几万字、甚至十几万字的网页原文（Raw Content）。
但是，**大模型的 Context Window (上下文窗口) 是有限的**！而且网页里充斥着广告、页脚和废话。如果一股脑把这些内容塞给 LLM，不仅费用昂贵，还会导致模型“注意力丢失”（Lost in the Middle）。

怎么办？这就要请出大名鼎鼎的 **RAG (检索增强生成) 与重排过滤 (Rerank)** 技术了。
